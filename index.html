能力对比

传统 PM vs AI PM

五个维度的差异，帮你清楚看到需要补足的能力缺口。

传统产品经理
功能清单驱动。PRD 写清楚每个按钮、每个交互状态，需求评审后进入开发。
AI 产品经理
能力边界驱动。PRD 描述期望的输入输出范围，定义 Eval 标准，接受输出的非确定性。
要点核心转变：从「定义确定的行为」到「定义可接受的行为范围」。
传统产品经理
QA 测试通过率、Bug 数、功能完整度。验收标准是二元的：有 or 没有。
AI 产品经理
Eval 框架驱动。准确率、相关性、安全性、延迟、成本，多维度持续评估，永不「完成」。
要点AI 产品没有「上线即完成」，模型迭代 = 产品迭代，Eval 是持续过程。
传统产品经理
设计确定的交互流程，用户走固定路径，每个页面状态可预期。
AI 产品经理
设计弹性交互，接受不确定性。设计反馈闭环让用户纠正错误，把「不完美」变成「可迭代」。
要点AI UX 的核心不是完美，而是「出错时优雅地恢复」。
传统产品经理
理解前后端 API、数据库结构，能和工程师讨论技术方案可行性。
AI 产品经理
理解模型能力边界、Prompt 工程、RAG 架构、Agent 设计模式，能和算法工程师讨论模型选型。
要点不需要会写训练代码，但必须理解「为什么模型在这里表现不好」。
传统产品经理
数据是运营分析工具——DAU、留存、转化漏斗，用于验证产品决策。
AI 产品经理
数据是产品本身——用户反馈是训练信号，bad case 是 Eval 数据集，数据飞轮驱动产品进化。
要点AI PM 最容易被低估的能力：设计数据飞轮，让产品越用越好。
避坑指南

新人最常见的四个误区

这些坑，每一条都有无数人踩过。提前知道，少走三个月弯路。

01
「我要先学会写代码再学 AI PM」
✗ 误区：把学编程当成前置条件，花 3 个月学 Python，还没碰到真正的 AI PM 知识。
✓ 正解：AI PM 的核心能力是产品判断 + Eval 思维 + Agent 设计，不是写代码。能看懂 Python 示例就够用了。直接开始学 Prompt 工程，在实践中学编程。
02
「学完理论就够了，项目以后再说」
✗ 误区：只看书和课程，不做实战项目。面试时只能说「我学过」，拿不出任何作品。
✓ 正解：项目经验是 AI PM 求职的硬通货。Phase 3 的实战项目不是可选项——它是你简历上最闪亮的部分。学完 Phase 2 就立刻开始做项目。
03
「Prompt 工程就是写几句话」
✗ 误区：以为 Prompt 就是「帮我写个文案」，随便写写就行。
✓ 正解：Prompt 工程是一套系统方法论——系统指令设计、few-shot 策略、思维链触发、输出格式控制、Eval 驱动迭代。这是 AI PM 的基本功，值得花 2 周以上深入练习。
04
「AI PM 就是给 AI 产品写 PRD」
✗ 误区：用传统 PM 的思路做 AI 产品——列功能清单、画流程图、定义每个交互状态。
✓ 正解：AI PM 的核心转变是从「定义确定行为」到「定义可接受的行为范围」。你需要设计 Eval 标准、接受非确定性输出、建数据飞轮。PRD 的写法完全不同——看 Compare 章节就懂了。

面试真题

AI PM 高频面试题

这些是真实面试中出现频率最高的问题。每道题附带了答题思路框架，帮你快速建立回答结构。

Agent 设计
设计一个客服 Agent，它需要能查订单、改地址、处理退款。你会怎么设计 Tool Use 和 Agent Loop？
思路先画 Agent 架构图（感知→工具选择→执行→观察→循环），再定义每个 Tool 的输入输出 schema，最后设计 fallback 策略（什么时候转人工）。重点讲清楚「多轮对话的状态管理」。
架构设计：用户消息进入后，先过意图识别（小模型/规则），判断属于查询类还是操作类。查询类直接调用对应 Tool 返回结果；操作类需要先确认再执行。

Tool 定义：① query_order(order_id) → 返回订单状态、金额、商品列表；② update_address(order_id, new_addr) → 需要用户确认后执行；③ process_refund(order_id, reason) → 金额超过阈值需人工审批。

Agent Loop：感知（接收消息）→ 思考（LLM 判断需要调哪个 Tool）→ 行动（执行 Tool）→ 观察（读取返回结果）→ 循环（如果需要多步则继续）。关键：每步都要记录到 context，避免重复调用。

Fallback 策略：连续 2 次工具调用失败 → 转人工；退款金额 >500 元 → 人工审批；用户情绪检测到负面 → 转人工并标记优先级。

加分项：提到用 Checkpointing 保存对话状态，用户中途离开再回来能接续；提到 Eval 框架评估 Agent 在 100 个测试用例上的表现。
Eval 方法
你的 AI 客服上线后准确率只有 75%，你会怎么系统性地提升？
思路分三步：1) 建 Eval 数据集，按错误类型分类（幻觉/拒答/理解错误）；2) 分析 Top 3 错误类型的根因（Prompt/数据/模型）；3) 针对性优化并重新评估。强调「数据驱动的迭代」。
第一步：建立 Eval 数据集
从线上日志中抽取 500 条 bad case，按错误类型标注：幻觉（编造不存在的信息）35%、拒答（该回答但不回答）25%、理解错误（答非所问）20%、格式问题 15%、其他 5%。

第二步：根因分析
幻觉 35% → 根因：知识库覆盖不足，模型在知识盲区编造答案。方案：加 RAG，对高频问题补充知识库文档。
拒答 25% → 根因：System Prompt 过于保守，安全策略误伤。方案：调整 Prompt 边界，区分「不能回答」和「不需要回答」。
理解错误 20% → 根因：用户口语化表达，模型没正确理解意图。方案：加 Few-shot 示例覆盖常见口语表达。

第三步：迭代与验证
每次修改后跑 Eval 数据集，对比修改前后准确率。目标：幻觉降到 10% 以下，整体准确率提升到 88%+。上线后持续监控，建立数据飞轮——用户反馈自动进入下一轮 Eval 数据集。

加分项：提到「不要只看整体准确率」，要分场景看（退款场景准确率可能比查询低 15%）；提到成本控制（不是所有 bad case 都值得修，按影响面排序）。
产品策略
RAG 和微调，什么场景选哪个？如果一个客户问「我的产品文档有 10 万页，怎么做 AI 问答」，你怎么回答？
思路核心判断：知识更新频率。频繁更新→RAG（改数据库不改模型）；稳定领域+特定格式→微调。10 万页文档→RAG + 分块策略 + 向量检索，不建议微调（成本高、更新难）。
决策框架：
选 RAG 当：知识需要频繁更新、需要溯源（标注答案来自哪份文档）、数据量大且持续增长、预算有限。
选微调当：需要特定输出风格/格式、领域术语密集且稳定、推理速度要求高（微调后小模型可能够用）、有足够的标注数据。

10 万页文档的具体方案：
1. 分块策略：按章节/段落分块，每块 500-1000 Token，保留上下文重叠 100 Token。不要按固定字数切——会切断语义。
2. 向量化：用 Embedding 模型（如 text-embedding-3-small）对每个块生成向量，存入向量数据库（Pinecone / Milvus / Chroma）。
3. 检索策略：用户提问 → 向量检索 Top 5 相关块 → 拼 Prompt → LLM 生成回答。加 Rerank 提升精度。
4. 为什么不微调：10 万页 ≈ 1.5 亿 Token，微调成本高且文档更新后需要重新训练。RAG 只需更新数据库。

加分项：提到混合方案——RAG 为主 + 微调一个小模型做意图分类/路由；提到 Eval（建 100 个标准 Q&A 对，评估检索召回率和回答准确率）；提到成本估算（每千次查询的 Token 成本）。
Agent 设计
你的 Agent 延迟太高（平均 15 秒），用户抱怨。你会从哪些维度优化？
思路四个方向：1) 模型层——换小模型做路由，大模型做核心；2) Prompt 层——减少 Token、缓存 system prompt；3) 架构层——并行工具调用、流式输出；4) 产品层——加 loading 动画、分步展示结果。关键是先测量再优化。
模型层：用更小的模型做路由 / 分类，只在核心步骤调用大模型；开启流式输出（streaming），让用户先看到部分结果而不是干等。

Prompt 层：压缩 system prompt、减少 few-shot 示例长度、去掉冗余指令；把可并行的子任务拆成独立 prompt 并发调用。

架构层：工具调用改并行（同时查订单 + 查库存，而非串行）；加结果缓存与语义缓存，相同问题不重复算；预取下一步可能需要的数据；用轻量 planner 减少规划步数。

产品层：骨架屏 + 渐进式渲染，先出结构再填内容；乐观 UI（先显示「正在处理」）；把长任务拆成可感知的子步骤并展示进度条。

加分项：给出「延迟预算拆解」——规划 2s / 检索 5s / 生成 8s，定位瓶颈在哪一段；用 trace（如 LangSmith）逐段监控，优先优化最慢的那一段而非整体。
产品策略
AI 产品和传统产品最大的区别是什么？用你自己的话解释。
思路三个核心差异：1) 确定性→概率性（接受非确定性输出）；2) 功能验收→持续 Eval（永不「完成」）；3) 数据是运营工具→数据是产品本身（数据飞轮）。用具体案例说明，不要背概念。
确定性 → 概率性：传统功能给定输入必有确定输出；AI 产品输出有方差，同一问题可能得到不同答案。PM 要接受「不完美但够用」，用置信度 / 兜底策略管理不确定性。

功能验收 → 持续 Eval：传统产品上线即验收完成；AI 产品上线只是开始，需持续评估质量、监控模型漂移，答案可能随模型升级变好或变差。

数据是产品本身：传统产品的数据是运营辅助；AI 产品的效果取决于数据飞轮——用户反馈、bad case、标注数据直接决定产品能力，数据闭环即核心竞争力。

加分项：补充「安全护栏与人在回路」——高风险操作（退款、删除）必须人工确认；可解释性不是技术点缀而是用户信任机制。用「传统 PM 管功能，AI PM 管概率」一句话收尾。
Eval 方法
什么是「幻觉」？你的产品里如何检测和降低幻觉？
思路幻觉 = LLM 生成看似合理但事实错误的内容。检测：自动（事实一致性校验）+ 人工（bad case 标注）。降低：RAG 提供事实依据、Prompt 约束（「不确定时说不知道」）、输出后校验。产品设计上让用户能纠错。
定义：幻觉 = 模型生成看似合理但事实错误、或根本不存在的内容（张冠李戴、编造来源、虚构数据）。

检测：自动——事实一致性打分、检索对齐度校验、NLI 自然语言推断；人工——抽样评测 + bad case 标注，建立幻觉率基线。

降低：RAG 基于可信知识库生成，让答案有出处；Prompt 约束「不确定时说不知道 / 必须引用来源」；用户纠错闭环——让真人能一键标记错误并回流训练。

加分项：按场景设分级容忍度（闲聊可宽松、医疗 / 金融要严格）；建幻觉率 dashboard 持续监控，而不是上线一次就完事。
技术协作
你的算法工程师说「这个效果做不出来，模型能力不够」。你怎么判断他说的是真的还是推脱？
思路三步验证：1) 要具体的 Eval 数据（准确率多少？在什么测试集上？）；2) 对比竞品/开源方案（别人做到了吗？）；3) 拆解问题（是模型能力问题还是 Prompt/架构问题？）。AI PM 不需要会写代码，但必须能看懂 Eval 结果。
要 Eval 数据：不让对方用「感觉不行」搪塞，要求量化指标（准确率多少？在什么测试集上？和 baseline 差多少？）。AI PM 不写代码，但必须看得懂评测结果。

对比竞品 / 开源：业界或竞品做到没？差距在哪？如果别人用类似方案做到了，那「做不出来」往往是范围或数据问题，不是能力问题。

拆解问题：把「做不出来」拆成子问题（是模型能力、数据质量、还是架构设计？），定位真瓶颈。很多「做不到」其实是「没拆对」。

加分项：设 1–2 天的 spike 小实验验证可行性，用数据决策而非立场；约定「做不到的判定标准」写入需求文档，避免反复扯皮。
产品策略
给你 3 个月和 2 个工程师，做一个 AI 产品 MVP。你会选什么方向？为什么？
思路考察产品判断力。好的回答：选一个「AI 能做到、传统方案做不到」的场景（而不是用 AI 重复造轮子），有明确的 Eval 标准，MVP 范围克制（1 个核心功能 > 5 个半成品），有商业化路径。避免选太泛的方向。
选 AI 独有价值：挑「AI 能做到、传统方案做不到 / 做不划算」的场景，而不是用 AI 重造一个表单。差异化才是 MVP 的意义。

明确 Eval：能定义可量化的成功标准（准确率、满意度、节省时长），否则无法判断是否做对了，也无法迭代。

克制范围：MVP 只验证一个核心假设，1 个功能 > 5 个半成品；先跑通主链路再补边角。

商业化：选有付费意愿、价值清晰场景，而非技术 demo；想清楚谁买单、为什么买单。

加分项：用「一周出可用雏形 + 数据验证」的节奏替代「一次到位」；先拿真实用户跑通再谈扩展，避免闭门造车三个月做出没人要的东西。
术语词典

AI PM 黑话翻译器

看到不认识的术语？来这里查。点击展开完整词典。

大语言模型LLM
用海量文本训练的 AI 模型，能理解和生成自然语言。代表：GPT-4、Claude、Gemini。PM 需要知道的是：它能做什么、不能做什么、为什么有时会「胡说」。
检索增强生成RAG
先从知识库检索相关文档，再让 LLM 基于文档生成回答。解决了 LLM 「知识截止」和「幻觉」问题。类比：开卷考试 vs 闭卷考试。
评估框架Eval
系统性地衡量 AI 产品输出质量的方法。包括：定义评估维度（准确率/相关性/安全性）、收集测试用例、跑分对比。AI PM 的核心技能之一。
模型上下文协议MCP
Anthropic 提出的开放标准，让 AI Agent 能连接外部工具和数据源。类比：AI 世界的 USB 接口——统一了 Agent 和工具之间的连接方式。
智能体Agent
能自主规划、调用工具、多步推理完成任务的 AI 系统。不只是「问答」，而是「做事」。核心机制：Agent Loop（感知→思考→行动→观察→循环）。
工具调用Tool Use
让 LLM 调用外部函数/API 的能力。比如查天气、搜数据库、发邮件。是 Agent 「做事」的基础——光会说不行，得能动手。
思维链Chain-of-Thought
让 LLM 在给出答案前先「展示推理过程」的技术。类似让学生写解题步骤而不是只写答案。能显著提升复杂推理任务的准确率。
少样本学习Few-shot
在 Prompt 中给 LLM 几个示例，让它理解你想要的输出格式和风格。类比：给新员工看几个之前的案例，他就能照着做。
微调Fine-tuning
在预训练模型基础上，用特定领域数据继续训练，让模型在某个方向更强。PM 需要判断：什么时候用 Prompt 工程（便宜），什么时候该微调（贵但效果好）。
幻觉Hallucination
LLM 生成看似合理但实际错误的内容。是 AI 产品最大的风险之一。PM 的核心职责：设计产品机制让用户能发现和纠正幻觉，而不是假装它不存在。
检查点Checkpointing
在 Agent 执行过程中保存中间状态，支持回退和恢复。类比：游戏的存档点。让长流程任务不会因为中途出错就全部重来。
人机协作Human-in-the-loop
在 AI 自动化流程中插入人工审核/决策节点。关键设计：什么时候该让人介入？介入成本 vs 错误成本如何权衡？这是 AI PM 的核心设计决策。
TokenToken
LLM 处理文本的最小单位，约 0.75 个英文单词或 0.5 个中文字。API 按 Token 计费——PM 必须理解 Token 成本如何影响产品设计（比如要不要缓存、要不要截断）。
温度Temperature
控制 LLM 输出随机性的参数。温度=0 时输出确定（适合数据提取），温度=1 时更创造性（适合写文案）。PM 需要根据场景选择合适的温度。
